## Scope Notes

- **Object-targeted spells** (`"object"` JSON, ROM `TAR_OBJ_INV`) and the object branches of `TAR_OBJ_CHAR_OFF` / `TAR_OBJ_CHAR_DEF` are now routed by `do_cast` (CAST-004/005/006, fixed 2.11.52). The spell handlers that accept `Object` targets (`identify`, `enchant_armor`, `enchant_weapon`, `fireproof`, `create_water`, `detect_poison`, `recharge`) receive the resolved object from `do_cast` via `spell_func(char, target_obj)`. Handlers that only accept `Character` targets (e.g. `bless`, `curse`, `poison`, `remove_curse`, `invisibility`) will raise on the Object — the fix routes correctly; handler-side Object branches are future parity work for the per-spell audit.
- **CAST-007 ordering note (ROM `check_killer` → charm gate interaction)**: ROM `do_cast` calls `check_killer` before the `AFF_CHARM` master gate (src/magic.c:406 then :408-412). For a charmed PC caster attacking a PC, `check_killer` enters the `AFF_CHARM` branch (lines 1248-1260) and calls `stop_follower(ch)`, which strips the charm affect and clears `ch->master`. The charm gate then sees `!IS_AFFECTED(ch, AFF_CHARM)` and does not block. For a charmed PC attacking an NPC (including their NPC master), `check_killer` returns early at line 1241 (`IS_NPC(resolved_victim)`) without reaching the charm branch, so charm remains intact and the charm gate blocks the cast. For NPC casters, both `is_safe`/`is_safe_spell` and `check_killer` are skipped (`!IS_NPC(ch)` guard), but the charm gate still fires (it is outside the `!IS_NPC` block in ROM). The Python implementation mirrors this ordering exactly.
