- **Status**: ✅ AUDITED 100% — all 7 gaps closed, including canonical `weapon_table`
- **Date**: 2026-05-14
- **Source**: `src/const.c` (ROM 2.4b6, 1936 lines, 16 top-level data tables)

  - `mud/models/constants.py` (LIQUID_TABLE, ATTACK_TABLE, INT_APP_LEARN, ItemType, …)

  - `mud/world/movement.py` (str_app[].carry column inline)
  - `mud/commands/equipment.py` (str_app[].wield column inline)

`const.c` is a pure data file — no functions. The audit walks each table, locates its Python counterpart (or marks it missing), and verifies that every consumer of the table's columns reads ROM-equivalent values. The CRITICAL gaps below are not "this table isn't ported"; they are "this table's stat-bonus column is silently absent from a behavioral macro consumer (combat / advancement) in Python."

## Phase 2 — Verification (load-bearing tables)

The 13 already-AUDITED rows above were checked by walking each row of the ROM table against the Python value, and (for the app-tables) by checking every ROM consumer against its Python equivalent. Per-row content for the small ✅ tables (`item_table`, `wiznet_table`, `attack_table`, `liq_table`, `int_app`) was diffed manually and matches.

### `str_app[26]` (ROM 728-755)

ROM consumers of `str_app`:

| ROM site | Column | Python equivalent | Status |
|----------|--------|-------------------|--------|
| `merc.h:2107-2108` `GET_HITROLL(ch)` macro — `ch->hitroll + str_app[STR].tohit` | `tohit` | `mud/math/stat_apps.py::get_hitroll` ports the macro; consumed at `mud/combat/engine.py:411,420` | ✅ CONST-002 FIXED |
| `merc.h:2109-2110` `GET_DAMROLL(ch)` macro — `ch->damroll + str_app[STR].todam` | `todam` | `mud/math/stat_apps.py::get_damroll` ports the macro; consumed at `mud/combat/engine.py:1189` | ✅ CONST-003 FIXED |
| `handler.c:923` `can_carry_w` — `str_app[STR].carry * 10 + level * 25` | `carry` | `mud/world/movement.py:115` — full ROM table inlined (`STR_APP_CARRY`), formula matches | ✅ AUDITED |
| `act_obj.c:1625`, `handler.c:1148` `can_carry_w` wield-weight gate — `str_app[STR].wield * 10` | `wield` | `mud/commands/equipment.py:21` — full ROM table inlined, formula matches | ✅ AUDITED |

The `tohit` and `todam` columns are not ported anywhere; combat reads the raw `hitroll`/`damroll` fields without the macro augmentation. This produces incorrect to-hit chances and damage for STR ≠ 8–13 (the ROM band where both columns are 0).

### `int_app[26]` (ROM 759-786)

Single column `learn`. Python `INT_APP_LEARN` (constants.py:232) — diffed all 26 values against ROM. Match. Consumer: `mud/skills/registry.py` `gain_practice_amount` reads `INT_APP_LEARN[INT]`. ROM `update.c` reads `int_app[INT].learn`. ✅ AUDITED.

### `wis_app[26]` (ROM 790-817)

Single column `practice`. Sole ROM consumer: `update.c:87` `advance_level` — `add_prac = wis_app[WIS].practice`, then `ch->practice += add_prac`. Python `mud/advancement.py:88-94` `advance_level`:

```python
char.practice += PRACTICES_PER_LEVEL  # constants.py PRACTICES_PER_LEVEL == 1
```

…has no WIS-driven addition. A WIS-25 character gains 1 practice per level instead of ROM's 1 + 5 = 6. **CONST-006 (IMPORTANT)**.

### `dex_app[26]` (ROM 821-848)

Single column `defensive`. Sole consumer: `merc.h:2104-2106` `GET_AC(ch,type)` macro — `ch->armor[type] + (IS_AWAKE(ch) ? dex_app[DEX].defensive : 0)`. Used at `fight.c:480-489` for combat AC and at `act_info.c:1594-1645` for the `score`/`look` AC display.

Python `mud/combat/engine.py:391` reads raw `victim.armor[ac_idx]` with no DEX adjustment; the score/look display likewise reads raw armor values. **CONST-004 (CRITICAL)** — DEX-3 has +40 AC penalty, DEX-25 has -120 AC bonus; missing this skews both combat hit math and the displayed AC tier.

### `con_app[26]` (ROM 851-878)

Two columns: `hitp` (used) and `shock` (defined but zero ROM call sites — verified by `grep -rn "con_app" src/`; only `update.c:78` references the struct, and it reads `.hitp`). Treat `shock` as dead data; not a gap.

Sole live consumer: `update.c:78` `advance_level` — per-level `max_hit` gain rolls `number_range(class_table[].hp_min, class_table[].hp_max) + con_app[CON].hitp`. Python `mud/advancement.py:91`:

```python
char.max_hit += hp  # hp from LEVEL_BONUS dict — class-only, no CON, no number_range
```

Python misses both the CON modifier and the per-level RNG roll. **CONST-005 (CRITICAL)** — high-CON players get short-changed HP each level; low-CON players also get the wrong floor. The missing `number_range` roll is a separate `class_table.hp_min/hp_max` consumer gap, but the data fix lands together with the CON port.

### `title_table` (ROM 421-721)

Used by `nanny.c` set-on-creation, `update.c` `gain_exp`/`advance_level` set-on-level, and `do_advance` for IMM forced level changes. Python now ports the full 4×61×2 ROM table in `mud/models/titles.py`, persists the level-1 default title during `create_character()`, and resets the class title during `advance_level()` using the same ROM data. `set_title` (`mud/commands/character.py:85`) remains the shared spacing helper. **CONST-001** is now closed.

### `weapon_table` (ROM 76-86)

ROM defines 8 rows of `(name, school_obj_vnum, WEAPON_X, &gsn_X)` and treats the
table as the canonical source for school-weapon selection and weapon-skill
lookup. Python previously duplicated parts of this across:

- `mud/combat/engine.py` (`WEAPON_SKILL_BY_TYPE`)
- `mud/account/account_service.py` (`_WEAPON_VNUMS`)
- `mud/models/character.py` (`_STARTING_WEAPON_SKILL_BY_VNUM`)

Python now ports the canonical table to `mud/models/weapon_table.py` and routes
those consumers through shared helpers:

- `weapon_table_entry_by_name`
- `weapon_table_entry_by_school_vnum`
- `weapon_skill_name_for_type`
- `weapon_skill_name_for_school_vnum`

This closes the structural parity gap and fixes a real behavioral miss the
subset mappings hid: non-default ROM school weapons like `staff`, `axe`,
`flail`, `whip`, and `polearm` now resolve and seed learned weapon skill
through the same canonical mapping.

## Phase 4 — Closures

None this session. Five gaps (CONST-002..006) are **not** OLC-deferrable: they are behavioral combat/advancement divergences and per `AGENTS.md` ROM-parity rules they must close. They are sequenced as separate `/rom-gap-closer` invocations because each requires its own integration test and a small `c_compat`-clean port of the source-of-truth ROM table.

Recommended close order:

1. **CONST-002, CONST-003, CONST-004** as a triplet — they all need the same primitive: a ROM-faithful `str_app`/`dex_app` table module + `get_hitroll(ch)` / `get_damroll(ch)` / `get_ac(ch, type)` accessor functions in `mud/models/character.py` (or a new `mud/math/stat_apps.py`). One closer per gap, one commit per gap, but the table import is shared.
2. **CONST-005** — port `con_app` table, rewrite `advance_level` to roll `number_range(class.hp_min, class.hp_max) + con_app[CON].hitp` (touches `class_table` HP fields too — verify they're ported on `mud/models/classes.py:ClassType`).
3. **CONST-006** — port `wis_app` (1-column), apply in `advance_level`.
4. **CONST-007** — ✅ closed 2026-05-14 by centralizing the ROM `weapon_table`.
