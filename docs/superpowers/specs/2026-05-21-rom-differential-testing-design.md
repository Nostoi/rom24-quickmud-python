# ROM Differential Testing Design

## Purpose

Define a repeatable way to compare ROM C behavior and QuickMUD behavior at the
right level of abstraction.

This spec exists because “audited” and “tests pass” were not enough to catch
live parity bugs in user-visible behavior. The new standard is to compare ROM
and Python outputs directly where practical, but to do so at the **behavior
boundary** rather than forcing every C function into an artificial harness.

## Problem Statement

The project previously relied too heavily on:

- structural audits
- helper-path tests
- smoke assertions such as `len(output) > 0` or broad substring checks

Those methods can miss real parity bugs when:

- the wrong field is rendered but the command still “works”
- login/session state differs at runtime but helper-path tests pass
- a ROM function is replicated internally but the observable output differs

We need a stricter comparison strategy that answers:

1. What should be compared directly function-to-function?
2. What should be compared scenario-to-scenario?
3. What should be treated as data parity rather than code parity?

## Design Principles

### 1. Compare at the highest trustworthy boundary

If the user experiences the behavior at a command/session boundary, compare at
that boundary.

Examples:
- `score`
- `look`
- login / reconnect prompts
- healer / shop output
- note/board flows

Do **not** prefer a lower-level helper comparison if it hides meaningful output
or state differences.

### 2. Use function-level differential tests only for pure or bounded logic

Function-to-function comparison is good when the ROM code is mostly determined
by explicit inputs and has limited global-state dependence.

Examples:
- age/hour formulas
- XP formulas
- AC description bucketing
- title lookup
- stat tables
- flag/table lookup behavior

Avoid forcing this style on functions that depend heavily on:
- descriptor state
- room/char linked lists
- global world registries
- socket delivery
- extensive side-effect chains

### 3. Runtime-path verification is mandatory for boundary-heavy surfaces

For `nanny.c`, `save.c`, `comm.c`, and similar flows, helper-only tests are not
closure evidence. Real server-path verification is required.

That means telnet/WebSocket tests for:
- initial login
- new character creation
- reconnect
- command output after login
- save/load cycles

### 4. Data parity is a first-class verification concern

If behavior depends on:
- area JSON
- DB rows
- title tables
- help files
- reset chains
- object/mobile prototypes

then the data must be checked alongside the code.

## Differential Testing Modes

### Mode A: Formula Differential Tests

**Use for:** pure mechanics and lookup behavior.

**Method:**
- read the ROM C formula or table
- encode exact expected outputs
- assert the Python function matches those outputs deterministically

**Examples:**
- `get_age`
- `xp_compute`
- `exp_per_level`
- AC description wording thresholds
- class/race/title lookups

**Strengths:**
- cheap
- deterministic
- high signal

**Weaknesses:**
- does not verify integration wiring

### Mode B: Transcript Differential Tests

**Use for:** user-visible command and session behavior.

**Method:**
- define a scripted scenario
- run it against ROM (or a trusted ROM transcript/fixture)
- run the equivalent scenario against QuickMUD
- normalize irrelevant formatting differences only
- compare ordered output

**Examples:**
- `score`
- login prompts
- `where`
- `who`
- `equipment`
- note writing / posting
- healer interaction

**Strengths:**
- validates what the player actually sees
- catches wording/order/state bugs

**Weaknesses:**
- higher setup cost
- transcript normalization must be disciplined

### Mode C: Scenario-State Differential Tests

**Use for:** flows where exact text is less important than exact state.

**Method:**
- run identical scenario steps
- compare end-state and key transitions

**Examples:**
- save → reload
- death → corpse/inventory/room state
- combat round progression
- mob wander/scavenge effects

**Strengths:**
- useful where output is noisy but state is clear

**Weaknesses:**
- can miss wording/prompt issues if used alone

## Normalization Rules

Transcript comparison should normalize only what is clearly non-semantic:

- timestamp prefixes generated by the Python harness
- transport-specific wrappers (`type: prompt`, JSON envelopes)
- ANSI/telnet framing when the comparison target is textual content

Do **not** normalize:
- wording
- line order
- punctuation
- spacing that is visible to the player
- prompt text
- class/race/title labels

If normalization would hide the bug, it is too aggressive.

## Suggested Artifact Layout

### Golden transcript fixtures

Store under a dedicated test-fixture tree, for example:

- `tests/fixtures/rom_transcripts/`

Organize by surface:

- `tests/fixtures/rom_transcripts/act_info/score_level1_mage_elf.txt`
- `tests/fixtures/rom_transcripts/nanny/new_character_login_prompts.txt`

### Differential tests

Prefer dedicated focused test files:

- `tests/test_act_info_transcript_parity.py`
- `tests/test_nanny_transcript_parity.py`
- `tests/test_save_boundary_parity.py`

## Decision Rule: Which mode to use?

Use this rule per gap or surface:

- **Pure formula or table?** → Mode A
- **Player-visible text/session flow?** → Mode B
- **Stateful gameplay flow with noisy text?** → Mode C

Many surfaces should use more than one mode.

Example:
- `score`
  - Mode A for age/AC/title helper expectations
  - Mode B for final output transcript

## Closure Criteria

A surface should only be called “verified” when:

1. ROM source was read
2. the right differential mode was chosen
3. the comparison is exact enough to fail on meaningful drift
4. the real runtime path was exercised if the surface is boundary-heavy
5. data dependencies were checked if they influence the behavior

## Non-Goals

This spec does **not** require:

- compiling and embedding every ROM C function into Python tests
- reproducing every internal C helper as a callable standalone artifact
- comparing implementation internals line-by-line

The goal is **behavioral evidence**, not performative similarity.

## Recommendation for the Next Agent

Start by applying this spec to:

1. `src/act_info.c`
2. `src/nanny.c`
3. `src/save.c`

Build the first transcript/fixture infrastructure only as needed for those
surfaces. Do not over-generalize the harness before the first three slices are
working.
